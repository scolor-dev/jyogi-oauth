jyogi-oauth 設計書

概要

部内向け OAuth 2.0 / OpenID Connect 認可基盤。認可サーバー・リソースサーバー・フロントエンドを1台の Linux サーバーに Docker Compose で構築し、将来的にサーバー分離が可能な設計とする。

OAuth 2.0 / OIDC フロー（Authorization Code + PKCE）

1
Client → Go
GET /oauth/authorize?response_type=code&client_id=xxx&redirect_uri=...&scope=read write&state=random&code_challenge=xxx&code_challenge_method=S256
2
Go → Browser
ログイン画面 (/login) へリダイレクト
3
Member → Go
認証情報を送信（ID/パスワード）
4
Go → Browser
同意画面 (/consent) を表示
5
Member → Go
同意を送信
6
Go → Redis
認可コードを保存（TTL: 600秒）
7
Go → Client
redirect_uri に認可コード付きでリダイレクト: ?code=auth_code&state=random
8
Client → Go
POST /oauth/token — grant_type=authorization_code&code=...&redirect_uri=...&client_id=...&code_verifier=...
9
Go → Redis
認可コードを検証・削除 → PKCE 検証 → JWT 署名（15分）→ リフレッシュトークン保存（7日）
10
Go → Client
トークンレスポンス: { access_token(JWT), refresh_token, id_token(openid時), token_type, expires_in: 900, scope }
11
Client → Rust
GET /api/resource — Authorization: Bearer <access_token>
12
Rust → PostgreSQL
JWT 署名検証 + exp チェック（Redis参照なし）→ リソース取得
13
Rust → Client
リソースレスポンス

データ設計

Redis（一時データ）

キー値TTL用途
auth:code:{認可コード}{client_id, member_id, scope, code_challenge, redirect_uri}600秒認可コード
auth:refresh:{hash}{member_id, client_id, scope}604800秒リフレッシュトークン
auth:session:{id}{member_id, oauth_params}86400秒メンバーセッション
auth:member_refreshes:{member_id}SET of refresh hashesなしメンバー別失効管理
ratelimit:*カウント60〜300秒Rate Limit

アクセストークンは JWT（TTL: 15分）のため Redis に保存しない。失効制御はリフレッシュトークンの削除で対応。

ページ必要ロール機能

ロールシステム

ロール権限
memberプロフィール編集、連携アプリ管理、自分のクライアント管理
moderatormember + メンバー管理
admin全操作（メンバー・クライアント・スコープ・システム設定）

root メンバー（username: root / password: 65536 / role: admin）はDB初期化時に自動作成される。
